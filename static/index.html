Schema & Architecture

How multi-tenant HR data isolation works

🏗️ Multi-Tenant HR Architecture

Why Multi-Tenancy in HR? — Every company's employee data (salaries, attendance, leaves) is extremely sensitive. In a SaaS like BambooHR or Gusto, each company must never see another company's data.


Row-Level Isolation — Every table has an org_id column. A single middleware extracts the X-Org-Id header and all queries filter by it.


How it works:

  1. Frontend sends X-Org-Id: 1 (TechVista) or X-Org-Id: 2 (Bloom Health)
  2. Middleware validates & sets request.state.org_id
  3. Every query: SELECT * FROM employees WHERE org_id = :org_id
  4. TechVista's salaries are invisible to Bloom Health — complete isolation!

Used by: BambooHR, Gusto, Rippling, Workday, and every HR SaaS product.

Database Schema

Table Tenant Key Description
organizations Root table Company registry — each row is one tenant company
employees org_id → organizations.id Employee directory, salaries, departments — scoped per company
attendance org_id → organizations.id Daily check-in/check-out records — scoped per company
leave_requests org_id → organizations.id Leave applications with approval status — scoped per company
